# AI-driven modular kit-bashing in Unreal Engine 5: a complete concept-to-scene pipeline

**An AI agent can now propose and spawn modular sci-fi layouts in UE5 via MCP tools, but the most effective pipeline is a layered hybrid: vision models interpret design intent, LLMs generate approximate grid layouts in JSON, constraint solvers (WFC) enforce structural validity, and MCP tools execute placement — with the human designer retaining final authority over composition, lighting, and narrative flow.** This approach bridges a critical gap: no existing system directly converts concept art into modular tile-based level layouts, yet the component technologies are mature enough to assemble a working pipeline today. The key enabling insight is that Unreal Marketplace demo scenes are machine-readable design grammars — scannable via MCP to bootstrap the AI's understanding of any kit.

## The mesh catalog that makes AI spatial reasoning possible

An AI agent making layout decisions needs two things simultaneously: natural-language understanding of what a piece *is* (semantic), and exact numeric data for where to place it (geometric). The minimum viable catalog format must bridge these worlds in a single JSON structure.

**Per-piece metadata falls into five layers.** The identity layer carries the UE5 asset path (`/Game/Kits/SciFi/SM_Wall_Straight`) and a human-readable description ("5m straight wall panel with ribbed metal surface and conduit runs"). The spatial layer records dimensions in centimeters, bounding box min/max, pivot offset, and grid snap interval — for a 500cm grid kit, all major dimensions should be multiples of 500. The connectivity layer defines sockets: named connection points with positions, outward-facing directions, and type labels (e.g., `wall_edge`, `floor_edge`, `door_frame`) that specify which socket types are compatible. The adjacency layer stores WFC-compatible rules: for each of the six cardinal directions, which other piece IDs may legally neighbor this piece. The semantic layer carries category (`wall`, `floor`, `corner`, `door`), tags (`structural`, `exterior`, `hero_piece`), visual style notes, and a spawn weight controlling how commonly the piece should appear.

The most battle-tested schema for adjacency rules comes from the **DeBroglie** WFC library's JSON format, which supports tile symmetry types (X for full rotational symmetry, I for 180°, T, L, and backslash for various partial symmetries), adjacency pairs per direction, path constraints ensuring connectivity, border constraints, and frequency multipliers. Adapting this for 3D modular pieces means extending the six-direction adjacency model (±X, ±Y, ±Z) with socket-type matching rather than simple tile-ID matching, since a wall's left edge socket can connect to any piece exposing a compatible `wall_edge` socket regardless of the specific mesh.

**The LLM consumes the catalog differently than the engine does.** For the LLM, the critical fields are `display_name`, `description`, `category`, `dimensions_cm`, `function` (natural-language role description), and the connectivity rules. For UE5 spawning via MCP, the critical fields are `asset_path`, `pivot_offset_cm`, and the exact socket positions. A well-designed catalog serves both consumers from a single source of truth. Game studios like Bethesda (documented extensively in Joel Burgess's GDC talks on Skyrim and Fallout 4 modular design) maintain similar internal documentation — standardized naming conventions, grid-unit specifications, kit category breakdowns (walls, floors, ceilings, corners, doorways, stairs, T-junctions, transition pieces) — though historically in spreadsheets and wiki pages rather than machine-readable JSON.

## Scanning demo scenes extracts the design grammar automatically

Most Unreal Marketplace asset packs ship with artist-authored preview levels demonstrating how pieces fit together. These demo scenes encode a **design grammar** — the rules governing which pieces connect, at what rotations, in what spatial patterns — that can be reverse-engineered programmatically.

Multiple MCP server implementations now exist for UE5. The **chongdashu/unreal-mcp** project (1,200+ GitHub stars) provides actor enumeration, property queries, and transform read/write via a Python MCP server communicating over TCP to a C++ plugin. **Flopperam's unreal-engine-mcp** (evolved into "Flop Agent" v0.6) adds advanced capabilities including physics simulation, Blueprint manipulation, and material systems. **GenOrca's unreal-mcp** contributes surface-placement raycasting, asset search/filtering, and generic property access. All follow the same architecture: AI client → MCP protocol → Python server → TCP socket → C++ UE5 plugin.

**The scanning pipeline works in five stages.** First, enumerate all actors via `get_actors_in_level`, extracting name, class, mesh asset path, transform (position, rotation, scale), bounding box, and tags for every StaticMeshActor. Second, preprocess by filtering to structural pieces (excluding lights, cameras, post-process volumes), detecting the grid size as the greatest common divisor of position deltas between same-type pieces, snapping positions to the detected grid, and quantizing rotations to 90° increments. Third, build an adjacency graph where nodes are actors and edges connect pieces within 1.0–1.5× grid distance that share a connection face. Fourth, run connected component analysis on this graph to identify spatial clusters — each connected component represents a discrete architectural unit (a room, corridor segment, or junction). Fifth, classify each cluster by archetype using geometric heuristics.

**Cluster classification relies on simple but effective heuristics.** Compute the bounding box aspect ratio and count external connection openings (edges where the cluster connects to other clusters or has unmatched sockets). A cluster with aspect ratio >4:1 and exactly two openings at opposite ends is a **corridor**. Three openings indicate a **T-junction**; four openings, a **crossroads**. An aspect ratio <2:1 with larger area and variable openings is a **room**. A single opening marks a **dead end**. Two openings with a 90° bend form an **L-corner**. These heuristics correctly classify the overwhelming majority of standard modular compositions.

An alternative to connected-component analysis is **DBSCAN clustering** with epsilon set to 1.5× grid size and minPts of 3–5, which handles noise (isolated decorative props) gracefully and discovers clusters of arbitrary shape. For multi-scale analysis — grouping individual walls into rooms, rooms into wings, wings into buildings — hierarchical clustering with average linkage provides a dendrogram that can be cut at different distance thresholds.

Each extracted template stores pieces with relative transforms (positions normalized to the cluster centroid), connection points at cluster boundaries with typed sockets and compatibility lists, the archetype classification, and metadata pointing back to the source demo level. The **Dungeon Architect** plugin's snap-connection model validates this approach: it uses categorized modules (Room, Corridor, TreasureRoom) with connection points at door entries/exits, placing door meshes at used connections and wall meshes at unused ones.

## Vision models can propose layouts but need constraint solvers to fix them

The research is unambiguous on one point: **no existing system directly converts concept art into structured modular tile-based level layouts.** This gap represents the pipeline's most novel and most fragile component.

What *is* feasible, backed by published systems, is a two-phase approach. In the first phase, a VLM analyzes the concept image to extract semantic content — room types, spatial relationships, approximate proportions, mood, and architectural features — and outputs a structured JSON description. In the second phase, an LLM uses chain-of-thought prompting to convert that semantic description into grid coordinates, and a constraint solver refines the result to satisfy modular kit rules.

**HouseLLM** (arXiv 2411.12279) demonstrates the most directly relevant pattern: an LLM generates initial room layouts in JSON format using CoT prompting (room-by-room sequential placement with center coordinates, width, and height), then a conditional diffusion model refines geometry. **ChatHouseDiffusion** (arXiv 2410.11908) shows GPT-4-turbo excelling at room type and location recognition from natural language but struggling with precise sizing. **Chat2Layout** handles irregular floor shapes and multi-room setups via sequential generation with grid-based placement algorithms. **Text-to-Layout** (arXiv 2509.00543) builds a full pipeline from natural language to JSON wall/door/furniture coordinates to Autodesk Revit models.

**Prompting strategies matter enormously for spatial output quality.** The Chain-of-Symbol (CoS) technique replaces natural language spatial descriptions with condensed symbolic representations and achieves up to **60.8% accuracy improvement** (31.8% → 92.6%) on spatial reasoning benchmarks for GPT-3.5-Turbo while reducing token usage by 65.8%. Visualization-of-Thought (VoT) prompting yields 27% accuracy boosts over standard chain-of-thought. Room-by-room sequential generation with explicit coordinate tracking is more reliable than generating entire layouts at once. Few-shot examples showing the kit catalog and example layouts are critical for generation quality.

However, VLM spatial reasoning has hard limits. The paper "Stuck in the Matrix" (arXiv 2510.20198) demonstrates that **LLM performance drops 42.7% on average (up to 84%) as grid size increases**. Models handle 5×5 grids reasonably but lose coherence at larger scales. They struggle with coordinate identification, distance computation, and maintaining geometric consistency across many cells. Concept art introduces additional challenges: perspective distortion requires spatial transformation to extract top-down layouts, artistic ambiguity leads to hallucinated structure, and stylized images defeat reliable scale estimation.

The practical implication is clear: **target small grids for LLM layout generation** (10×10 cells or fewer per generation pass), use symbolic/structured representations over natural language, provide the full kit catalog as context with few-shot examples, and always post-process with a constraint solver that enforces grid alignment, socket compatibility, and connectivity.

## WFC and PCG complement AI layout rather than compete with it

Wave Function Collapse, UE5's PCG framework, and Houdini procedural tools each address different layers of the level generation problem. The optimal pipeline uses all of them.

**WFC excels at local coherence** — ensuring every tile transition looks correct — but has zero understanding of global structure, narrative, or function. It operates by iteratively collapsing cells with minimum entropy and propagating adjacency constraints to neighbors. Commercial games using WFC include Bad North (island terrain), Caves of Qud (ruins), Townscaper (cities), and Epic's Matrix Awakens demo (city generation). UE5 ships with an experimental built-in WFC plugin, and commercial alternatives exist on Fab (LebeDevUE's "Wave Function Collapse 3D"). WFC's key limitation for this pipeline is that it requires exhaustive adjacency rule authoring — but these rules are exactly what the demo-scene scanning pipeline extracts automatically.

**UE5's PCG framework** (production-ready since UE5.7 with GPU execution for ~2× speedup) is designed for scatter-based placement — distributing vegetation, rocks, and props across surfaces — not constraint-based modular snapping. It lacks built-in adjacency or socket matching. However, its custom node extensibility and integration with World Partition streaming make it valuable for the detail-filling pass after structural layout is established.

**Houdini Engine** provides the most powerful procedural modeling capabilities via HDAs (Houdini Digital Assets) that expose promoted parameters to Unreal users. Existing HDAs for modular level generation include Radu Cius's "Procedural Level Design" tool and the "Modular Procedural Dungeon" pack. Houdini's strength is encoding complex constraints via VEX/Python, but it requires a separate license ($269/year Indie, $4,495/year commercial) and has a steep learning curve.

The **CG-WFC** hybrid (Cyclic-Graph + Wave Function Collapse) points toward the right architecture: a two-layer system where Layer 1 uses graph grammar rules for mission structure (lock-and-key progression, narrative pacing) and Layer 2 uses WFC for spatial realization. Replacing Layer 1's hand-authored graph grammar with an LLM that interprets design intent from text or concept art creates the hybrid pipeline:

- **LLM/VLM layer**: Interprets concept art or text description → outputs abstract layout spec (room graph with types, sizes, relationships)
- **WFC/constraint layer**: Takes the abstract spec → produces valid tile placements satisfying adjacency rules extracted from demo scenes
- **PCG/detail layer**: Populates structural layout with environmental props, vegetation, lighting fixtures
- **MCP execution layer**: Translates placement data into `spawn_actor` calls in UE5

**DreamGarden** (CHI 2025 Best Paper by Sam Earle et al.) validates this layered approach: it uses an LLM-driven planner that breaks down a single high-level prompt into hierarchical action plans distributed across specialized submodules for Unreal Engine environment development — the first automated game prototyping tool for UE demonstrated in a peer-reviewed publication.

## Where AI accelerates and where humans remain essential

The evidence from industry adoption and academic studies draws a consistent boundary between AI and human strengths in level design.

**AI provides its greatest value in initial population speed and variant generation.** Promethean AI (founded by former Naughty Dog art director Andrew Maximov) reports up to **10× faster production times** for environment building by automating setup tasks. The system understands semantic context — suggesting kitchen items in kitchens, workshop items in garages — and can rearrange layouts to new constraints while preserving spatial relationships between objects. The Morai Maker study (ACM CHI 2019) found that designers valued AI as a brainstorming partner that "gets me moving when I'm stuck." King studios has deployed AI level-design tools for casual and puzzle games where constraint satisfaction and variant generation are more important than bespoke artistic composition.

**Humans remain irreplaceable for composition, emotional resonance, and narrative-driven design.** The professional consensus, reflected in the 2025 GDC State of the Game Industry report (where **82% of game developers** said human designers are essential to creating engaging games), frames AI as "co-pilot, not autopilot." AI-generated content consistently lacks what professionals describe as "artistic depth and intentionality." Lighting decisions require holistic understanding of mood and player psychology. Environmental storytelling — the subtle cues that convey history, danger, or safety — depends on intentional imperfection: sagging roofs, overgrown paths, asymmetric wear patterns. No current AI system reasons about these qualities.

For the modular kit-bashing pipeline specifically, the optimal division of labor is:

- **AI handles**: First-pass structural layout, grid-compliant placement, connectivity verification, variant generation for rapid iteration, populating repetitive areas, and enforcing constraint satisfaction
- **Human handles**: Overall composition and focal points, lighting design and mood, narrative flow through spaces, hero moment placement, micro-detail and intentional imperfection, final aesthetic polish

## A practical five-phase onboarding pipeline for new asset packs

Converting a freshly downloaded Unreal Marketplace pack into an AI-ready catalog requires systematic extraction and validation. The pipeline below takes roughly 2–4 hours for a ~20-piece kit, with most time spent on manual semantic annotation.

**Phase 1 — Import and inventory** (automated, ~5 minutes). Run a Python script using `unreal.AssetRegistryHelpers.get_asset_registry()` to enumerate all Static Meshes, Materials, and Blueprints. Record asset paths, naming patterns, and counts. Open the demo map to understand the pack's intended composition.

**Phase 2 — Grid and convention analysis** (automated, ~10 minutes). For each Static Mesh, read bounding box dimensions via `mesh.get_bounding_box()`. Compute the GCD of all major dimensions to identify the base grid unit. For a 500cm grid kit, verify all structural pieces have dimensions that are multiples of 500cm (or 250cm for half-pieces). Check pivot locations: floor pieces should have pivots at a bottom corner, wall pieces at a bottom edge. Flag any inconsistencies. Analyze the demo scene by extracting all actor positions and computing position deltas to confirm the grid — rotations should cluster at 0°, 90°, 180°, 270°.

**Phase 3 — Catalog generation** (semi-automated, ~1 hour). Build the JSON catalog with automated fields (asset path, dimensions, bounding box, pivot, material slots) and manually curated fields (category, description, function, visual style, tags). Extract connectivity rules by analyzing the demo scene's adjacency graph: for each pair of pieces that appear adjacent in the demo, record the relative transform as a valid connection. Infer socket types from recurring connection patterns.

**Phase 4 — Validation and testing** (automated, ~30 minutes). Programmatically spawn each mesh at origin, verifying no errors. Spawn all observed adjacent pairs from the demo scene, verifying gap-free alignment with no z-fighting. Test all rotation variants. Validate that collision meshes exist. Confirm grid compliance across all spawn positions.

**Phase 5 — AI-ready tagging and testing** (manual + automated, ~1–2 hours). Add semantic annotations for functional role, placement rules, context tags, and density hints. Run a simple AI-driven layout test: prompt the LLM with the catalog and a description like "generate a straight corridor with one side room" and execute the resulting spawn commands via MCP. Verify the output, iterate on catalog descriptions and rules until the AI consistently produces valid placements.

The critical insight is that **most of Phase 3's connectivity rules can be extracted automatically from the demo scene** rather than authored manually. The demo scene is the artist's statement of how the kit works — mining it captures the design grammar implicitly.

## Conclusion

The concept-to-scene pipeline for AI-driven modular kit-bashing is viable today, though it requires assembling components from multiple domains rather than using any single off-the-shelf tool. The most important architectural decision is the layered hybrid: VLM for semantic interpretation, LLM for approximate layout generation in JSON, WFC/constraint solver for structural validation using rules extracted from demo scenes, and MCP for execution in UE5. The two highest-risk components are VLM spatial reasoning (which degrades severely beyond ~10×10 grids) and the concept-art-to-structure gap (which no published system has fully bridged). The two highest-value components are automated demo-scene scanning (which eliminates days of manual catalog authoring) and the WFC integration (which guarantees every AI-proposed piece placement is structurally valid). Studios adopting this pipeline should expect AI to eliminate 60–80% of the time spent on initial layout population while accepting that final composition, lighting, and narrative polish will remain human tasks for the foreseeable future.
