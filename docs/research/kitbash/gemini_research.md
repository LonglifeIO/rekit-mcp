# Concept-to-Scene Pipeline for Modular Kit-Bashing

Designing an AI-driven pipeline to turn concept art or text into a grid-based layout of modular pieces requires rich data about each piece and the kit’s “grammar.”  In practice, each asset must carry information about its size, pivot/origin, and how it connects to neighbors.  Modular kits are typically built with *fixed dimensions and connection rules*【23†L1090-L1098】.  For example, wall segments, floor tiles, and corner pieces are modeled so their edges align on a consistent grid.  Thus, an AI or procedural system needs each piece’s geometry (bounding box or footprint) and local pivot offset so it knows how to place it precisely on the grid.  It also needs to know which faces or edges of that mesh are *open* or *connector* faces where other pieces can snap on.  Equivalently, each piece should be tagged with metadata or attachment “ports” (e.g. left/right/front/back) indicating how it can join others.  In short, the AI catalog should include: the mesh reference, its global dimensions, the pivot point location, and a description of valid snap-points or edge types.  (These might be explicit coordinates or simply semantic tags like “wall-end,” “doorway,” etc.)  This lets the AI spawn the mesh and align it seamlessly with neighbors.  

With this data, a minimal *mesh catalog* can bridge visual understanding and spawning.  A practical JSON schema might include fields such as: 

- **id/name**: an identifier or asset path for the piece.  
- **mesh reference**: pointer to the 3D model or asset.  
- **dimensions**: the footprint size (e.g. 500×500×H cm) and bounding box.  
- **pivot offset**: the origin point relative to the mesh geometry (important for snapping on the grid).  
- **snap points/connectors**: a list of faces or edge indices where this piece can attach (possibly with orientation flags). For instance, a straight wall piece might list “front” and “back” connectors, a corner piece has “left” and “forward” connectors, etc.  
- **tags/categories**: semantic labels (e.g. “wall”, “floor”, “door”, “corridor”) to inform style or adjacency logic.  
- **thumbnail or text description** (optional): a small image or string describing appearance, which could help a vision-language model match pieces to a concept image.  

Including a small icon or CLIP embedding could help an AI “understand” the style, but the core needed fields are the spatial and connectivity data.  (For comparison, 2D sprite atlas formats often include a `pivot` field【26†L257-L265】; here the pivot is 3D and the “atlas” is the entire level grid.)  In effect, this catalog lets a generator or LLM treat each modular piece like a “building block” with known shape and connectors.  

## Extracting Templates from Demo Scenes

Most Unreal asset packs include a *preview level* that demonstrates how pieces fit together.  We can leverage these human-authored scenes as examples of the kit’s design grammar.  Practically, one would load each preview/demo map and programmatically enumerate all placed actors.  Using tools like the Unreal MCP interface or Python editor scripts, one can call `get_actors_in_level` (or similar APIs) to list every actor in the scene【58†L448-L450】.  For each actor that uses a modular mesh from the kit, retrieve its static mesh identifier and transform (location and rotation).  This yields a dataset of “mesh X at grid position Y.”  

From this data, we can deduce snapping conventions and common assemblies.  For instance, by looking at which meshes lie adjacent on the grid (e.g. one grid unit apart and aligned), we can infer which faces connect.  If a wall piece is placed at (0,0) and another at (500,0) with matching faces, that records a valid “wall-to-wall” connection at that orientation.  Clustering the placed pieces by spatial region can reveal repeated structures: a set of meshes forming a rectangular *room*, or a linear sequence forming a *corridor*.  One could algorithmically detect such groupings (e.g. flood-fill contiguous grid positions) and label them as archetypal layouts (“square room of size 2×2”, “corner bend in hallway”, etc.).  

These reverse-engineered templates become higher-level building blocks.  For example, the system might store that “RoomTypeA” consists of 4 walls and a floor in a 2×2 arrangement, or that “T-junction corridor” comprises one long corridor piece with side walls branching.  By scanning many demo scenes, the AI learns which piece combinations occur frequently.  This yields a *per-pack catalog* of sample configurations: compact rooms, door transitions, junctions, etc., which the AI can later reuse or adapt.  (No published tool specifically does exactly this, but the approach is similar to building a 3D *scene graph* of objects and relationships【30†L59-L68】, except here the objects are known modular parts.)  

## Related AI Layout and PCG Techniques

There is active research on generating layouts from high-level inputs.  Recent work shows that language models can *plan* visuals and spatial layouts.  For instance, **LayoutGPT** demonstrates that an LLM can convert text instructions into plausible 2D/3D layouts【45†L53-L62】.  In LayoutGPT, the model learned to arrange objects given natural language prompts, even handling numerical spatial constraints.  Similarly, **ChatDesign** (a 2024 CAADRIA paper) uses a chat-based LLM loop to iteratively refine a floor plan based on text instructions【38†L288-L297】.  ChatDesign maps free-text goals into core operations (locate, move, add, delete) to build the plan.  These examples suggest that an AI can indeed translate a user’s description into a structured layout: for example, a prompt like “build a large hall with side alcoves” could yield a room plus adjoining smaller rooms.  

No direct academic example yet takes a *sketch* and assigns specific modular pieces, but the principles apply. A multimodal model (e.g. GPT-4V) could, in theory, interpret a concept image and output a top-down floorplan. If we supply the model with the kit catalog (perhaps as a reference list or embedded tokens for each piece), we could prompt it: *“Given this sketch of a sci-fi corridor, and this list of available wall/floor/door pieces, place pieces on a 500cm grid.”* The model would match shapes in the sketch to pieces (e.g. recognizing a doorway gap as a ‘door piece’) and emit a grid map or actor list. Systems like LayoutGPT hint that with enough examples or in-context images, LLMs can learn spatial composition【45†L53-L62】.  In practice, one might generate synthetic image-to-floorplan training data from the demo scenes (render top-down views of example rooms), then fine-tune or few-shot a vision-language model to map images to grid instructions.  

In summary, prior work indicates feasibility: LLMs can learn a “grammar” of scene assembly from examples【45†L53-L62】, and interactive systems can iteratively interpret design intent【38†L288-L297】.  The novel twist here is constraining the output to our specific modular kit – but by deriving the catalog and templates from demo levels, we give the AI concrete vocabulary and patterns to use.  

## Vision Models for Layout Generation

Modern vision-language models have strong scene-understanding capabilities. For example, tools like GPT-4V can caption and analyze images of interiors, and research shows GPT-4V can identify objects and spatial relations within photos【30†L59-L68】【45†L53-L62】.  If a concept sketch or painting is available, a vision model could potentially segment it into regions (rooms, hallways) and infer where openings (doors, windows) are. Given our kit catalog, the model could then label each region with the best-fit modular pieces. For instance, it might see a rectangular room and assign 4 wall meshes plus 1 floor mesh in that footprint.  

This is an emerging area; no turnkey solution exists yet. However, the success of LayoutGPT in interpreting text-to-layout suggests similar capacity when combining vision and language. A practical approach would be to prompt GPT-4V with the concept image and a description of available kit pieces, asking it to output a grid-based layout (e.g. coordinates and piece names). Its “limitations” would depend on how easily it can reason in grid units and recall our catalog. It may need careful prompting or fine-tuning on examples of images paired with explicit layouts. In any case, with a well-structured mesh catalog and a good prompt engineering, a vision-capable model could draft a first-pass plan from a concept illustration or reference. The final accuracy would likely be lower than in-text prompts, but it could capture the user’s intent (e.g. "two long halls intersecting") and leverage the kit’s grammar to fill in details.  

## Strengths and Limits of AI-Generated Layouts

AI-driven layout excels at speed and variety but struggles with fine-grained aesthetic polish.  In practice, procedural generators (or LLMs) can rapidly explore many variants of a design, but “the levels it produces won’t be as appealing as if a creative person made them by hand”【48†L69-L77】.  In other words, a human designer is better at giving a scene visual interest, appropriate focal points, and avoiding obvious repetition.  AI can handle the heavy lifting of rough blockout – for example, quickly filling a dungeon with corridors, rooms, and basic structural rules – but a human eye often “wins” when it comes to final styling, narrative coherence, or subtle composition.  

A common pragmatic approach is *hybrid*: use AI or procedural methods to generate a layout base, then have designers tweak it. As one developer advises, let the generator create an initial level and then “manually refine for visual flair”【48†L103-L112】. This marries the strengths of both: the generator saves time and prevents designer fatigue by producing many fast variations, while the designer ensures the final output meets the artistic vision.  In our pipeline, this means the AI-driven proposal is a starting framework: the user can then adjust piece placements, add detail meshes or adjust lighting in the editor. In summary, AI helps most in early-stage iteration and bulk arrangement, whereas humans do the final pattern-breaking and polish【48†L69-L77】【48†L103-L112】.  

## PCG Frameworks vs AI-Driven Layout

Unreal’s built-in Procedural Content Generation (PCG) framework offers a powerful alternative or complement to AI. PCG uses node graphs to spawn and filter points in space, then instantiate assets; it excels at rule-based, parametric generation【50†L17-L25】.  For example, a PCG graph could flood a volume with floor tiles at a regular interval or lay out corridors by following a grid algorithm. Because PCG is deterministic (given seeds and parameters), it’s easier to guarantee geometric consistency (no overlapping walls, fully filled floors) without needing a learned model.  

By contrast, AI/ML-based layout (LLMs or learned planners) can incorporate higher-level patterns learned from data. For instance, AI might recall that this kit often has alternating windowed and solid wall segments in a corridor, or that rooms usually have a centerpiece. This knowledge isn’t explicit in PCG rules unless handcrafted. In practice, the two can complement each other. One could use PCG to enforce hard constraints (e.g. “cover this bounding box entirely with floor pieces, leave these grid lines for walls”) and then use AI to choose which wall piece variant goes on each side based on style. Or vice versa: use an LLM to propose a layout graph (room sizes, connectivity), then have a PCG graph realize it by placing meshes at the grid points.  

Notably, Epic’s documentation highlights PCG’s role: it enables “fast, iterative tools” for content of any complexity【50†L17-L25】. This suggests one could quickly prototype layouts with PCG nodes and then refine with AI prompts. In sum, for strict grid-based modular placement, PCG ensures precision and speed under fixed rules, while AI adds semantic design knowledge and variation. A balanced pipeline might use PCG for brute-force geometry and reserve AI for the creative patterning and high-level decisions.  

## Onboarding a New Asset Pack

When adding a new modular kit, an automated pipeline might proceed as follows: 

1. **Load Demo Level:** Import the pack and open its preview/demo level(s).  
2. **Enumerate Actors:** Use MCP or Unreal Python to call `get_actors_in_level` and gather each placed actor.  
3. **Filter by Kit:** Keep only actors using the new kit’s meshes.  
4. **Extract Data:** For each actor, record the mesh name, transform (grid-aligned position/rotation), and any tags. Also retrieve mesh bounds and pivot via `get_actor_transform` or `get_actor_details`【58†L448-L450】.  
5. **Infer Connectivity:** Analyze which meshes abut on the grid. For adjacent actors (within snapping tolerance), note their relative orientation. Mark these pairs as valid connections. This reveals the kit’s snapping logic and common adjacencies.  
6. **Cluster Compositions:** Group nearby actors into rooms or corridors. For each cluster, save the list of piece transforms as a “template composition” (e.g. a 3×3 room, a straight corridor segment, an L-shaped corner).  
7. **Build Catalog Entries:** Create JSON entries for each unique mesh (with dimensions, pivot, connectors) and for each identified template (with anchor points and variant parameters).  
8. **Test Spawning:** Use the MCP spawn commands (like `spawn_actor_by_class`) to place pieces using the new catalog in a test level, verifying they align correctly.  
9. **Iterate:** If some pieces appear misaligned (due to pivot quirks) or combinations were missed, adjust the catalog or run more sophisticated neighbor-detection until the templates replicate the demo.  

Once this data is compiled, the pack is “AI-ready”: an LLM can reference it to place pieces.  This automated scan of preview scenes jump-starts the AI’s understanding of the kit. No source directly describes this exact pipeline, but it builds on the MCP tools listed (e.g. actor queries【58†L448-L450】) and on the insight that demo maps encode design intent.  In summary, the pipeline is: *scan demos → extract meshes/transforms → infer snap rules and sample layouts → assemble a reference catalog → validate with spawn tests → use in AI composition.*

Finally, industry experience notes that procedurally generated levels need human refinement【48†L69-L77】【48†L103-L112】, suggesting the value of a hybrid workflow in a kit-bashing pipeline.
